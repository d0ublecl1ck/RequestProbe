# editorial-frontend-theme Specification

## ADDED Requirements
### Requirement: 主应用必须呈现黑白编辑式工作台壳层

桌面端主应用 MUST 采用深色导航壳层与浅色内容画布的组合，而不是统一的浅色玻璃背景。品牌强调色 MUST 谨慎使用，只在关键操作和重要状态中出现。

#### Scenario: 用户进入主界面

- **WHEN** 用户打开应用
- **THEN** 左侧导航区域 MUST 呈现深色沉浸式外壳
- **THEN** 右侧主内容区 MUST 使用浅色高可读画布承载正文信息
- **THEN** 品牌强调色 MUST NOT 作为大面积背景色铺满整个界面

### Requirement: 每个工作区页面必须提供统一的页面骨架

“字段探针”“资源监听”“设置”三个工作区 MUST 都提供页面标题、说明、主摘要区和至少一个清晰的正文区块，而不是只显示零散卡片。

#### Scenario: 切换到任一工作区

- **WHEN** 用户从侧边栏切换到任一工作区
- **THEN** 页面 MUST 先展示该工作区的标题与用途说明
- **THEN** 页面 MUST 提供一个可快速感知当前状态或主任务的摘要区
- **THEN** 页面正文 MUST 以分组区块方式组织表单、代码、表格或设置项

### Requirement: 工作区页面必须为关键交互提供明确反馈

所有工作区页面 MUST 为高频操作提供清晰的主次按钮层级、空状态说明和当前状态反馈，避免用户在长页面中丢失上下文。

#### Scenario: 页面存在尚未执行的主任务

- **WHEN** 用户首次进入字段探针、资源监听或设置页面
- **THEN** 页面 MUST 提供可理解的空状态或引导说明
- **THEN** 主操作按钮 MUST 与辅助按钮在视觉上明显区分

#### Scenario: 页面出现结果或状态变化

- **WHEN** 用户完成请求解析、启动监听任务或保存设置
- **THEN** 页面 MUST 在摘要区或结果区中反馈最新状态
- **THEN** 结果浏览区 MUST 保持清晰分组，而不是让新内容无序堆叠
