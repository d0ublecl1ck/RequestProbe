## Context

当前项目前端主要由单个应用壳层、左侧工作区导航和三个工作区页面组成。虽然页面划分已经具备基础结构，但全局仍使用暖色玻璃态背景、大圆角卡片和轻量工具风按钮，和 `DESIGN.md` 中强调的黑白对照、编辑式排版、极少量品牌红点缀、2px 精密圆角、强叙事分区完全不同。

这次不是单独微调某一个卡片，而是一次跨入口壳层、侧边栏、共享容器和三个页面内容区的前端重构。约束也很明确：

- 必须覆盖每个工作区页面，而不是只替换首页外观。
- 不能破坏现有 Wails 绑定、字段探针结果流和资源监听任务流。
- 需要继续兼顾窄窗口下的可用性，而不是只做大屏静态排版。

## Goals / Non-Goals

**Goals:**

- 建立可复用的 Ferrari 风格视觉 token 和组件皮肤，统一全局色彩、字重、间距、边框和状态反馈。
- 让应用壳层呈现“黑色导航 / 白色内容 / 深色沉浸区块”的节奏，而不是当前统一浅色玻璃风。
- 为字段探针、资源监听、设置三个页面分别建立清晰的信息架构和高频操作区。
- 保持现有业务逻辑、页面状态和后端调用不变，只重构前端呈现与交互组织。

**Non-Goals:**

- 不新增前端路由，不把当前工作区拆成多页面 URL。
- 不修改 Go、Python、Wails bindings 或业务接口。
- 不引入新的状态库、组件库或图片资源依赖。

## Decisions

### 1. 先重构全局 token，再改页面骨架

先在 `index.css` 中建立新的黑白中性色、品牌红、标签字重、2px 半径、分区背景和过渡规则，再让共享容器组件消费这些 token。这样三个页面不会各自散落写一套视觉规则，后续也更容易保持一致。

备选方案是逐页局部覆盖旧样式，但那会保留大量玻璃态遗留规则，最终很容易出现设计语言混杂。

### 2. 侧边栏采用“深色品牌带 + 编辑式导航卡片”

保留左侧导航这一现有交互模型，但把它升级为深色英雄区 + 页面切换区的组合结构。顶部负责品牌存在感和工作台定位，底部提供明确的工作区切换与当前态高亮。这样既不违背现有导航规格，也能把 `DESIGN.md` 的沉浸式入口气质落到桌面应用中。

### 3. 页面内部使用“英雄摘要 + 编辑内容区块”的统一骨架

三个工作区都会采用统一框架：

```text
+-----------------------------------------------------------+
| Dark hero summary: title / status / primary action        |
+-----------------------------------------------------------+
| Light editorial sections: forms / tables / code / config  |
+-----------------------------------------------------------+
```

字段探针页面会突出输入与结果双栏；资源监听页面会突出任务状态、控制区与资源/请求浏览；设置页会突出当前目录、说明和操作按钮。这样用户在不同工作区切换时可以获得一致的认知模型。

### 4. 对高频交互增加更明确的状态反馈和可浏览性

字段探针的解析、测试、分析动作会通过更强的按钮层级、摘要卡片和结果标签来区分；资源监听页面会把运行状态、下载目录和任务控制从普通表单提升为可扫描的控制台区；设置页会把“当前值 / 默认值 / 立即操作”拆开，减少用户判断成本。

### 5. 保持现有页面切换状态实现，不引入新路由

继续使用 `activeWorkspaceTab` 和现有本地状态。此次重构重点是体验和结构，不把问题升级成路由改造，以降低回归风险。

## Risks / Trade-offs

- [视觉改动范围大] → 先抽象共享壳层和通用 section 样式，再分页面替换，避免样式碎片化。
- [深色外壳可能影响可读性] → 仅把导航和沉浸摘要区做深色，表单、表格、代码主体保持浅色阅读面。
- [窄窗口下双栏拥挤] → 继续保留现有响应式堆叠能力，并优先保证主操作按钮与关键结果不被压坏。
- [旧样式残留导致风格冲突] → 在共享容器和全局样式中统一替换核心视觉类，避免页面局部覆盖。

## Migration Plan

1. 先落全局 token、应用壳层和侧边栏。
2. 再重构字段探针页面，因为它的层级最复杂，能验证新骨架是否可承载长表单和长结果。
3. 接着改资源监听页面与设置页，使整体风格与交互一致。
4. 最后跑前端构建，修正样式残留和响应式问题。

## Open Questions

- 当前仓库没有 FerrariSans 或 Body-Font 资产，因此实现上将使用现有可用字体与更接近的字重、间距和大小去逼近设计语气，而不引入新的外部字体文件。
