## Context

当前字段探针页使用左右双栏布局。左侧承载请求输入、解析结果和操作按钮，右侧是一个带滚动容器的纵向结果列，依次展示 `Python 代码`、`请求测试结果`、`测试摘要`、`简化代码` 四张卡片。这样虽然实现简单，但用户在完成“测试请求”或“字段分析”后，需要滚动查找对应结果区域，流程引导不够明确。

这次变更只影响 React 前端，不涉及 Wails 绑定、Go 服务或 Python 逻辑。项目已经有通用 `Tabs` 组件，因此应优先复用现有 UI 原语，避免新增状态库或路由变化。

## Goals / Non-Goals

**Goals:**

- 将右侧区改为单一 Tabs 容器，减少滚动查找成本。
- 默认聚焦 `Python 代码`，符合解析请求后的首要阅读路径。
- 在字段分析完成后自动切换到 `测试摘要`，让用户立即看到结论。
- 尽量保持现有卡片内容、复制按钮、表格和内部子 Tabs 结构不变，降低回归风险。

**Non-Goals:**

- 不重构左侧输入区和字段配置区布局。
- 不改动字段分析接口、测试请求接口和结果数据结构。
- 不新增“记住上次页签”“根据结果智能选择多个页签”等更复杂的导航策略。

## Decisions

### 1. 右侧区使用单一受控 Tabs 状态

新增一个受控状态表示右侧当前页签，默认值固定为 `python-code`。右侧所有内容都挂载在这个 Tabs 下，而不是继续以四张卡片顺序渲染。

选择理由：

- 能精确实现“默认固定打开 Python 代码”和“字段分析完成后自动跳转测试摘要”。
- 与现有项目里工作区级 Tabs、响应详情 Tabs 的使用方式一致，维护成本低。

备选方案：

- 使用锚点滚动定位到对应卡片：需要保留四张卡片堆叠结构，状态不可见，体验不如显式页签。
- 使用折叠面板：仍会保留纵向浏览心智，且不如 Tabs 明确。

### 2. 每个 Tab 面板继续复用现有卡片内容

每个页签面板内部仍渲染原来的卡片头、复制按钮、表格和空状态提示，只是放入对应的 `TabsContent` 中，不改业务逻辑。

选择理由：

- 变化集中在布局容器和少量状态流，能最大限度减少 UI 回归。
- 保持现有视觉样式，不需要额外设计新组件。

### 3. 自动跳转只绑定在字段分析成功完成时机

当字段分析动作成功拿到摘要结果后，前端主动将右侧当前页签切换到 `test-summary`。普通解析请求、单次测试请求和清空结果不触发自动跳转。

选择理由：

- 与用户要求精确一致。
- 避免单次测试或清空时产生意外跳页，打断当前阅读位置。

## Risks / Trade-offs

- [右侧内容仍然较长] → Tabs 只解决信息分栏，不解决单个面板内部长度问题；保留每个面板现有滚动能力。
- [自动跳转时机绑错] → 只在字段分析成功设置结果后切换，避免在开始分析或失败时误跳。
- [现有内部 Tabs 嵌套后样式异常] → 复用现有 `Tabs` 组件并做最小结构调整，验证请求结果内的子 Tabs 仍正常工作。

## Migration Plan

1. 创建 OpenSpec 变更并补齐 proposal、design、tasks、delta spec。
2. 在字段探针页面右侧区加入新的受控 Tabs 状态与面板结构。
3. 在字段分析成功回调后切换到 `测试摘要` 页签。
4. 运行前端构建验证，确认样式、受控状态和现有交互未被破坏。

回滚策略：

- 若该改动引起布局或交互问题，可回退为原有纵向卡片结构，不影响底层请求与分析逻辑。
