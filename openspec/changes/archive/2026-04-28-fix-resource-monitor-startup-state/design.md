## Context

当前资源监听页面把 `task?.resources || []` 和 `task?.requests || []` 直接作为依赖使用。无任务时这些表达式会在每次渲染产生新的空数组，配合同步 `selectedIds` 的 `useEffect`，会在空状态下持续触发 `setState`，形成无限更新。与此同时，Wails 调试态对前端地址使用自动推断，WebView 场景下会把 Vite HMR 的连接目标推导到不可解析主机或缺失端口。

## Goals / Non-Goals

**Goals:**
- 让资源监听页面在无任务时只表现为稳定的“未开始”状态
- 消除资源监听页面的无限更新告警
- 让 `wails dev` 下的前端资源加载和 HMR WebSocket 使用固定、可解析的本地地址

**Non-Goals:**
- 不调整资源监听任务的后端生命周期语义
- 不引入新的状态管理库或改造整体页面结构
- 不处理与本次问题无关的生产打包行为

## Decisions

### 决策 1：在前端对任务对象做显式归一化

资源监听页面增加轻量归一化逻辑，把缺失关键字段的空对象视为“无任务”，而不是直接进入“有任务但字段为空”的分支。

选择原因：
- 可以兼容 Wails 绑定在不同环境下对空指针返回值的差异
- 变更范围小，只影响资源监听页面

备选方案：
- 在 Go 侧额外包装返回值。未采用，因为当前问题首先表现为前端初始化不稳，而且前端仍需要防御事件流中的空对象

### 决策 2：为资源与请求列表提供稳定空数组引用

无任务时改用模块级常量空数组，避免依赖项在每次渲染中变化；同步勾选状态时仅在内容真正变化时返回新对象。

选择原因：
- 直接消除 `useEffect` 无限循环根因
- 不引入额外库，符合现有 React 写法

备选方案：
- 删除同步勾选状态的 effect。未采用，因为任务更新后仍需自动清理失效勾选项

### 决策 3：显式固定 Wails dev 与 Vite HMR 地址

将 `wails.json` 的前端开发服务器地址改成 `http://127.0.0.1:5173`，并在 Vite 中显式设置 `host`、`port`、`strictPort`、`hmr.host` 与 `hmr.clientPort`。

选择原因：
- 避免 WebView 环境对 `wails.localhost` 的解析差异
- 避免 HMR 在非浏览器标准来源下回退出 `undefined` 端口

备选方案：
- 保持自动探测，仅增加 Vite `hmr` 配置。未采用，因为 Wails 侧仍可能继续注入不稳定的开发服务器来源

### 决策 4：通过前端显式 favicon 资源消除浏览器默认根路径探测

在 `frontend/public/` 下提供 favicon，并在 `index.html` 中显式声明引用，让开发态和生产态都走同一套资源路径。

选择原因：
- 直接修复当前唯一剩余的 404 请求
- 兼容 Vite 的 `public` 资源约定，改动最小

备选方案：
- 忽略该 404。未采用，因为它会持续污染启动控制台，影响调试信号质量

## Risks / Trade-offs

- [风险] 固定 5173 端口可能与本机已有进程冲突 -> 缓解：启用 `strictPort`，让冲突在启动时显式暴露，而不是运行时构造错误 WebSocket
- [风险] 前端归一化过严可能误伤真实任务 -> 缓解：仅把缺少 `taskId`、`status`、`downloadDir`、`createdAt`、`updatedAt` 且没有资源/请求内容的对象视为空
