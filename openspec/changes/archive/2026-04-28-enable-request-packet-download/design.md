## Context

请求列表当前只保留展示所需的摘要字段，例如请求/响应头、请求体摘要和响应体摘要。这些字段足够列表展示，但不足以直接形成“可下载请求包”。要支持批量下载，必须在 Python worker 处理 `DataPacket` 时同步准备一个可序列化、可落盘的请求快照。

## Goals / Non-Goals

**Goals:**
- 让请求列表支持勾选、全选和批量下载
- 将请求包保存为结构化 JSON，便于后续检索和复现
- 复用当前任务目录，不引入额外保存配置

**Non-Goals:**
- 不把请求包重新转换为 curl、HAR 或 Postman 格式
- 不做请求包内容编辑或重放
- 不单独新增新的下载根目录设置

## Decisions

### 决策 1：请求包以 JSON 文件保存到 `requests/` 子目录

每个请求包落盘为一个独立 `.json` 文件，目录为 `<taskDownloadDir>/requests/`。

选择原因：
- 与资源文件同属一个任务目录，结构清晰
- 避免把 `.json` 请求包和真实资源文件混在同一层

### 决策 2：在捕获请求时同步构建下载载荷

worker 在 `handle_packet()` 中为每个请求记录生成 `_download_payload`，下载时直接写文件，而不是再依赖活动页面重新查询。

选择原因：
- 下载动作不再依赖页面当前状态
- 能覆盖失败请求和已结束任务中的历史记录

### 决策 3：二进制 body 统一转为 base64 结构

请求体或响应体如果是字节序列，则写成包含 `kind=base64`、`encoding=base64` 和 `data` 的结构；文本或 JSON 兼容值保持原样。

选择原因：
- 保证 JSON 文件始终可解析
- 避免二进制内容在 UTF-8 文本文件中损坏

## Risks / Trade-offs

- [风险] 大响应体会让内存中的请求记录变重
  - 缓解：当前实现仍然是单任务内存态，先以功能完整为主；后续若出现明显内存压力，再考虑为大 body 增加截断或延迟持久化策略
- [风险] 用户连续下载同一请求会产生重复写入
  - 缓解：沿用资源下载的策略，若目标文件已存在则标记为已下载并跳过
