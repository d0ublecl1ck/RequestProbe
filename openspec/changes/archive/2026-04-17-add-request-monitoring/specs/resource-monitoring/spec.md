## ADDED Requirements

### Requirement: 系统可以实时展示页面请求列表
系统 MUST 在现有资源监听页面中同时展示当前浏览器任务捕获到的页面请求列表。请求列表 MUST 至少包含请求方法、请求地址、资源类型、状态码、失败状态、首次命中时间和请求摘要，并且 MUST 在无需手动刷新的情况下实时更新。

#### Scenario: 成功请求进入列表
- **WHEN** 监听中的浏览器完成一个页面请求并拿到对应响应
- **THEN** 系统 MUST 将该请求记录加入当前任务的请求列表
- **THEN** 界面 MUST 实时展示该请求的方法、URL、资源类型和状态码

#### Scenario: 失败请求进入列表
- **WHEN** 监听中的浏览器出现一个加载失败的页面请求
- **THEN** 系统 MUST 将该请求记录加入当前任务的请求列表
- **THEN** 该记录 MUST 明确标记为失败
- **THEN** 界面 MUST 展示失败原因或错误文本，而不是静默忽略

### Requirement: 请求监听与资源监听必须共用同一个任务生命周期
系统 MUST 让请求监听与资源监听共用同一个浏览器任务，而不是要求用户分别启动两个独立任务。任务暂停、恢复和结束时，请求事件流与资源事件流 MUST 一起受控。

#### Scenario: 启动任务后同时观察两类数据
- **WHEN** 用户启动一个资源监听页面任务
- **THEN** 系统 MUST 在同一个浏览器会话中同时开始资源监听和请求监听
- **THEN** 当前任务返回数据 MUST 同时包含资源列表和请求列表

#### Scenario: 暂停任务时停止接收新请求
- **WHEN** 当前任务处于“监听中”且用户点击“停止监听”
- **THEN** 系统 MUST 停止接收新的请求事件
- **THEN** 系统 MUST 保留当前已捕获的请求列表和资源列表

#### Scenario: 恢复任务时继续接收新请求
- **WHEN** 当前任务处于“已暂停”且用户点击“继续监听”
- **THEN** 系统 MUST 在原浏览器实例上继续接收新的请求事件和资源事件
- **THEN** 恢复之前已经捕获的请求记录 MUST 保留
