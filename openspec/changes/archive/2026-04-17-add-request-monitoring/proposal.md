## Why

当前“资源监听”工作区只能展示命中的静态资源，无法回答“页面实际发了哪些请求、哪些请求失败、请求方法和状态码是什么”这类更基础的问题。用户已经明确希望同时监听页面资源和网络请求，因此需要把现有浏览器监听任务扩展为双通道观测，而不是只保留资源下载能力。

## What Changes

- 扩展现有资源监听任务，使同一个浏览器会话同时产出“资源命中事件”和“请求监听事件”。
- 在资源监听页面新增请求列表视图，实时展示请求方法、地址、资源类型、状态码、失败信息和请求摘要。
- 保持现有资源筛选、下载、打开目录、暂停恢复和任务生命周期行为不变，只增强任务返回的数据结构与界面展示。

## Capabilities

### New Capabilities

无

### Modified Capabilities

- `resource-monitoring`: 现有资源监听能力需要同时覆盖页面请求监听、请求列表展示和请求失败可见性。

## Impact

- Go 侧资源监听模型、事件分发和 Wails 绑定模型会新增请求记录字段。
- Python `resource_monitor_worker.py` 会从 DrissionPage 抓包结果中提取请求级信息并实时回传。
- 前端资源监听页面会新增请求列表视图与对应状态同步逻辑。
