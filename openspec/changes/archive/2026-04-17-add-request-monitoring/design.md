## Context

当前资源监听页面的后端链路是 `Wails App -> ResourceMonitorService -> Python worker -> DrissionPage listener`。这个链路已经能在 `Network.loadingFinished` 后拿到完整 `DataPacket`，并从响应体中筛选符合后缀的资源，但 worker 只把符合扩展名的资源转成 `resource_detected` 事件，导致请求级信息在 Python 层就被丢弃。

这意味着即使底层已经捕获到了请求方法、请求头、失败信息和状态码，前端仍然看不到任何请求时间线。新增“请求监听”最合理的做法不是新起一套浏览器任务，而是在现有 `DataPacket` 流上同时构建 `MonitoredRequest` 记录，并与资源记录共用同一个任务状态。

## Goals / Non-Goals

**Goals:**

- 在不新增第二个浏览器任务的前提下，让单个监听任务同时产出资源列表和请求列表。
- 让前端可以实时查看请求方法、URL、资源类型、状态码、失败原因和请求摘要。
- 保持现有资源筛选与下载工作流不变，并继续复用当前任务生命周期控制。

**Non-Goals:**

- 不实现 HAR 导出、请求重放、断点篡改、批量重发等 DevTools 级高级能力。
- 不新增请求过滤器、搜索器或复杂详情抽屉，本次只提供稳定的实时列表和基础字段。
- 不修改字段探针工作区，也不引入新的全局工作区入口。

## Decisions

### 1. 复用同一条抓包数据流，同时生成两类实体

Python worker 继续以 `DataPacket` 为唯一输入源，但不再只处理资源响应。每个完成或失败的 packet 都会先构建 `MonitoredRequest`，再按扩展名判断是否额外生成 `MonitoredResource`。

这样做的原因：

- 请求监听和资源监听天然属于同一浏览器会话，拆成两个任务会导致状态割裂。
- `DataPacket` 已经同时持有 request / response / fail_info / resourceType，重复抓包没有必要。
- 前后端只需要维护一份任务生命周期和一条事件总线。

```text
DrissionPage DataPacket
        |
        +--> MonitoredRequest  (全部完成/失败请求)
        |
        +--> MonitoredResource (仅命中所选后缀的响应)
```

### 2. 请求记录按时间线保留，不做内容去重

资源记录仍然按响应体哈希去重，因为下载目标本质上是“内容集合”。请求记录则按事件时间线保留，因为同一个 URL 的多次调用在调试中是有意义的，去重会直接损失观察价值。

实现上为每条请求生成独立 ID，并保留：

- 方法、URL、资源类型
- 状态码、失败标记、失败原因
- 请求头、响应头
- 请求体摘要、响应体摘要
- 首次命中时间

### 3. 在资源监听页面内部增加“资源 / 请求”子视图，而不是新增工作区

顶层工作区导航保持不变，资源监听页面内部通过局部 Tab 切换“命中资源列表”和“请求列表”。这样可以保证：

- 用户仍然把两类信息视为同一次监听页面任务的不同观察面。
- 不污染左侧全局导航。
- 复用现有任务状态卡和启动表单。

## Risks / Trade-offs

- [请求列表增长过快] → 本次先保留完整实时列表，但只展示摘要字段，避免首版就引入复杂虚拟滚动；如后续数据量过大，再单独加分页或上限裁剪。
- [抓包字段在不同响应类型下不稳定] → Python 侧统一做容错提取和摘要截断，不假设请求体或响应头一定存在。
- [Wails 模型变更导致前端类型漂移] → 通过重新生成 `frontend/wailsjs` 并跑前端构建校验生成绑定。

## Migration Plan

1. 新增 OpenSpec 变更与 delta spec，明确 `resource-monitoring` 已覆盖请求监听。
2. 扩展 Go 模型与 Python worker，先让后端事件流包含请求列表。
3. 更新前端资源监听页面，增加请求列表子视图并接入实时事件。
4. 重新生成 Wails 绑定并运行 Go / 前端校验。

## Open Questions

无。当前需求可以在现有任务模型内闭环完成。
