## Context

当前资源监听任务通过 `prepareDownloadDir(taskID)` 直接把下载目录创建在应用配置目录下的 `RequestProbe/downloads/<taskID>`。这个实现保证了任务隔离，但把“保存根目录”与“任务目录生成规则”耦合在一起，导致前端无法提供用户可配置的保存位置。

本次需求不是修改资源监听的核心监听行为，而是在不破坏当前任务隔离规则的前提下，引入一个可持久化的保存根目录设置，并新增设置页面作为承载入口。

## Goals / Non-Goals

**Goals:**

- 提供一个清晰的设置入口。
- 允许用户查看和修改资源监听默认保存根目录。
- 启动资源监听任务时，在设置中的根目录下创建 `<taskID>` 子目录。
- 保持现有资源监听任务、下载、打开目录等行为兼容。

**Non-Goals:**

- 不修改资源监听资源去重、暂停/恢复、下载逻辑。
- 不增加多项全局设置系统。
- 不支持为单个任务临时覆盖保存目录。

## Decisions

### 1. 将“保存根目录”与“任务目录”拆分

后端新增资源监听设置对象，仅保存默认根目录。实际任务目录仍在启动时由系统生成：

```text
settings.resourceMonitorSaveRoot
        +
      <taskID>
        =
task.downloadDir
```

这样既满足用户可配置，又保留每个任务隔离落盘的结构。

### 2. 使用应用私有配置文件持久化设置

资源监听设置会写入用户目录下的应用配置文件，而不是只存在前端内存里。这样应用重启后设置仍然存在。

### 3. 设置页作为独立工作区

项目当前使用左侧工作区切换而不是 React Router，因此“设置页面”实现为新的工作区 tab，避免引入额外路由层和状态复杂度。

```text
侧边栏
├── 字段探针
├── 资源监听
└── 设置
```

### 4. 目录选择由后端包装 Wails 系统对话框

前端不直接依赖 Wails dialog runtime，而是调用后端方法打开目录选择器。这样前端只消费统一的设置接口，绑定方式也更稳定。

## Risks / Trade-offs

- 如果用户把保存根目录设置到无权限位置，启动任务时可能失败；系统需要返回明确错误。
- 当前只支持“全局默认根目录”，不支持任务级覆盖；这是为了控制范围并保持设置模型简单。
- 设置页新增后，左侧导航信息量会增加一项，但这是符合用户预期的显式入口。

## Verification

1. 打开设置页时可以看到当前资源监听保存根目录。
2. 用户可以通过选择目录或手动输入并保存设置。
3. 启动新的资源监听任务后，任务下载目录应位于“保存根目录/<taskID>”。
4. 资源监听页展示的下载目录与实际任务目录保持一致。
5. 旧有资源监听流程不回归。
